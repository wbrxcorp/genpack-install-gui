# 検討メモ: GUI 起動不能時の TUI フォールバック

- 日付: 2026-07-17
- 状態: **保留**（実装しない。将来再開する場合の記録）

## 動機

- ディスプレイ未接続の機体・KMS でモードセットできない GPU・シリアルコンソールのみの SBC などでは、現状 GUI が起動時に即クラッシュし、systemd の `Restart=on-failure` による再起動ループになるだけで打つ手がない。
- root 処理は `InstallerBackend` trait で UI から分離済みのため、TUI フロントエンドを追加してもディスク列挙・イメージスキャン・インストール処理はそのまま再利用でき、追加コスト自体は「画面」だけで済む。

## 検討結果と保留理由

### 1. フォールバック判定が決定的にできない（保留の主因）

- `/dev/dri` の出現タイミングやディスプレイ（EDID）認識タイミングは起動ごとに揺れるため、OS 起動時に「GUI を起動できない」ことを決定的に判定する方法がない。
- 「`--wait-for-display` あり → GUI を無限待ち」「なし → 即 TUI フォールバック」という割り切りにすると、結局ユーザーが起動時に GRUB メニュー等でグラフィクスの有無を事前選択しなければならず、自動フォールバックの意味が薄れる。
- 「`/dev/dri` 自体がない → 即 TUI」「DRM はあるがコネクタ未接続 → 数秒のグレース待ち後 TUI」という段階案も検討したが、グレース時間は結局ヒューリスティックであり、遅いディスプレイ認識の実機が TUI に落ちる誤判定を排除できない。

### 2. TUI の出力先も非自明

- 「ディスプレイがないから GUI を起動できない」場面では、ローカル VT に TUI を出しても誰にも見えない。実際の読者はシリアルコンソール（または SSH）。
- カーネルが `console=` として認識している TTY に出すとしても、それが複数の場合がある。
- systemd ユニット側の設計（`StandardInput=tty` / `TTYPath=` またはシリアル用の別ユニット）とセットで決める必要があり、アプリ単体では完結しない。

### 3. UI 形式（これは結論あり）

- やるなら行指向の番号選択式プロンプトで始めるのが堅牢。dumb terminal・低ボーレートのシリアルでも壊れず、依存クレートも増えない。ウィザードは「ディスク選択 → イメージ選択 → オプション → 確認 → 進捗」と直線的なので行指向で十分表現できる。
- 機能的には既存のコマンドライン版 `genpack-install` と近くなるが、引数を知らないユーザーが対話だけでインストールを完遂できる導線としては別物であり、作る意味はある。

## 再開する場合の入口

- 自動フォールバックではなく、明示フラグ（例: `--text`）から始める。SSH 越しの手動インストールやテストにはこれだけでも有用で、判定問題（上記 1）と出力先問題（上記 2）を回避できる。
- 自動フォールバックはその後、配備側（systemd ユニット・ブートメニュー）の設計と合わせて改めて検討する。
